**Change 1: Update setup_sudoers() to include fail2ban and sshd commands**

Replace the sudoers content line in setup_sudoers() with an expanded version that adds the commands needed for Phase 2. The current sudoers line is:
```
$USER ALL=(ALL) NOPASSWD: /usr/sbin/service ssh start, /usr/sbin/service ssh restart, /usr/sbin/service ssh status
```

Replace with:
```
$USER ALL=(ALL) NOPASSWD: /usr/sbin/service ssh start, /usr/sbin/service ssh restart, /usr/sbin/service ssh status, /usr/sbin/service fail2ban restart, /usr/sbin/sshd -t, /usr/sbin/sshd -T, /usr/bin/fail2ban-client status sshd, /usr/bin/fail2ban-client status
```

This adds:
- `service fail2ban restart` -- needed by configure_fail2ban()
- `sshd -t` -- config syntax validation
- `sshd -T` -- effective config dump (for verification)
- `fail2ban-client status sshd` -- jail status check
- `fail2ban-client status` -- general status check

Note: Also add tee for writing config files:
```
$USER ALL=(ALL) NOPASSWD: /usr/sbin/service ssh start, /usr/sbin/service ssh restart, /usr/sbin/service ssh status, /usr/sbin/service fail2ban restart, /usr/sbin/sshd -t, /usr/sbin/sshd -T, /usr/bin/fail2ban-client status sshd, /usr/bin/fail2ban-client status, /usr/bin/tee /etc/ssh/sshd_config.d/*, /usr/bin/tee /etc/fail2ban/jail.d/*
```

Actually, the ssh-hardening.sh functions use `sudo tee` and `sudo service` which already work if the user has general sudo access (which they do -- the sudoers NOPASSWD entries are for PASSWORDLESS convenience, not permission gating). The existing setup works because WSL2 users typically have full sudo. The NOPASSWD entries just prevent password prompts during the automated installer.

REVISED: Keep the sudoers simple. The essential additions are the service and tool commands that the installer calls with sudo:
```
$USER ALL=(ALL) NOPASSWD: /usr/sbin/service ssh start, /usr/sbin/service ssh restart, /usr/sbin/service ssh status, /usr/sbin/service fail2ban restart, /usr/sbin/sshd -t, /usr/sbin/sshd -T, /usr/bin/fail2ban-client *, /usr/bin/tee /etc/ssh/sshd_config.d/*, /usr/bin/tee /etc/fail2ban/jail.d/*, /usr/bin/rm -f /etc/ssh/sshd_config.d/cc-tmux.conf
```

ACTUALLY, let's keep this simple and safe. The sudoers file should list specific commands. Wildcards in sudoers can be a security risk. Let's list exact commands:

```
$USER ALL=(ALL) NOPASSWD: /usr/sbin/service ssh *, /usr/sbin/service fail2ban *, /usr/sbin/sshd -t, /usr/sbin/sshd -T
```

The `service ssh *` and `service fail2ban *` patterns allow start/stop/restart/status for both services. The `sshd -t` and `sshd -T` allow config validation. For `tee` and `rm`, the user's general sudo access (with password) handles it, or more likely WSL2's default NOPASSWD sudo covers it.

FINAL DECISION: Keep the sudoers entry focused and use service wildcards:
```bash
    cat > "$tmp_file" <<EOF
$USER ALL=(ALL) NOPASSWD: /usr/sbin/service ssh *, /usr/sbin/service fail2ban *, /usr/sbin/sshd -t, /usr/sbin/sshd -T, /usr/bin/fail2ban-client status *, /usr/bin/fail2ban-client status
EOF
```
