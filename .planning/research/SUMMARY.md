# Research Summary: CC x TMUX V2

**Domain:** WSL2-based persistent terminal workspace with remote access
**Researched:** 2026-03-20
**Overall confidence:** HIGH

## Executive Summary

CC x TMUX V2 is a ground-up refactor of a working V1 proof-of-concept that enables Windows users to run Claude Code in persistent tmux sessions accessible from both PC and phone via SSH tunnel. The core technology choices (tmux + SSH + ngrok on WSL2) are sound and battle-tested. The V2 effort is primarily about robustness, security, and UX -- not about replacing the fundamental stack.

The most impactful improvement is switching from password-based SSH to Ed25519 key-based authentication. V1 exposes password auth through a public ngrok tunnel, which is a genuine security risk. Combined with fail2ban and a hardened sshd_config, V2 can offer defense-in-depth security that makes remote access safe by default.

The second major improvement is leveraging WSL2's systemd support (available since WSL 0.67.6) to properly manage SSH as a system service, replacing V1's fragile `sudo -n service ssh start` pattern. This also enables fail2ban and other services to run properly.

The tunnel layer should be architected as pluggable from the start. ngrok remains the correct default (lowest friction for beginners), but Tailscale is the clear upgrade path for users who want persistent addresses. The key insight from research is that Tailscale must run on the Windows host, NOT inside WSL2, due to MTU issues. This means the Tailscale setup path is fundamentally different from ngrok (which runs inside WSL2), reinforcing the need for a modular tunnel architecture.

## Key Findings

**Stack:** Bash 5.x + tmux 3.4+ + OpenSSH 9.x + ngrok (default) or Tailscale (upgrade) + fail2ban + jq + TPM with tmux-resurrect/continuum. All available via apt/snap on Ubuntu 22.04/24.04 in WSL2.

**Architecture:** Scripts organized under `~/.cc-tmux/` instead of scattered in `~/`. Pluggable tunnel providers via sourced bash modules. Config stored in a single `config.env` file generated by the installer. Auto-detection of device type (mobile/desktop) via tmux client-attached hook checking terminal width.

**Critical pitfall:** Tailscale cannot run inside WSL2 reliably (MTU 1280 issue, packet encapsulation failure). Must run on Windows host with port forwarding to WSL2's SSH. This is a non-obvious gotcha that would waste hours to debug without prior knowledge.

## Implications for Roadmap

Based on research, suggested phase structure:

1. **Foundation & Installer** - Build the idempotent installer with auto-detection, directory structure, and dependency management
   - Addresses: One-command install, auto-detection, error handling, uninstall
   - Avoids: V1's manual placeholder replacement, scattered home directory files

2. **SSH Hardening & Security** - Ed25519 key generation, hardened sshd_config, fail2ban, systemd integration
   - Addresses: Key-based auth, security defaults, service persistence
   - Avoids: Password auth exposure through public tunnel

3. **Tunnel Layer** - Pluggable tunnel architecture with ngrok default, Tailscale support
   - Addresses: Remote access, address persistence (Tailscale), auto-reconnection
   - Avoids: Hard-coding to single tunnel provider, ngrok free tier lock-in

4. **Workspace Management** - tmux configuration, session persistence (TPM + resurrect + continuum), project management, mobile auto-detection
   - Addresses: Add/remove projects, device-adaptive UI, session survival across reboots
   - Avoids: Manual mode toggle, editing shell scripts to manage projects

5. **Polish & Distribution** - Desktop shortcut automation, health check/diagnostics, self-update mechanism, README
   - Addresses: Windows integration, troubleshooting, user documentation
   - Avoids: Copy-paste heavy setup instructions

**Phase ordering rationale:**
- Foundation first because every subsequent phase depends on the directory structure and config system
- SSH before tunnels because the tunnel exposes SSH -- it must be secure before being exposed
- Tunnels before workspace because the tunnel is how mobile users access the workspace
- Workspace after infrastructure because tmux config depends on knowing the tunnel provider and device detection strategy
- Polish last because it wraps user-facing convenience around working internals

**Research flags for phases:**
- Phase 3 (Tunnels): Tailscale-on-Windows-host-with-WSL2 integration needs hands-on testing. The MTU workaround and port forwarding setup should be validated on real hardware.
- Phase 4 (Workspace): tmux `client-attached` hook for mobile detection needs testing with actual Termius on Android to confirm `#{client_width}` reports accurately.
- Phase 5 (Polish): Self-update mechanism needs design work -- git pull vs release download vs version check only.

## Gaps to Address

- Termius on Android: Does it correctly report terminal dimensions to tmux? Does it support Ed25519 key import? (Likely yes, but needs hands-on validation)
- Windows Task Scheduler: Can we auto-start the WSL2 workspace on Windows login without user interaction? Research was inconclusive -- may need phase-specific investigation.
- Self-update mechanism: How should a bash-only tool check for and apply updates? Git pull from the repo? Check GitHub releases API? This needs design during the Polish phase.
- Multiple WSL distros: What happens if the user has multiple WSL2 distros? The installer should detect and handle this gracefully.
