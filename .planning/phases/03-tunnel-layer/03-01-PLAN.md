Create `lib/tunnel/ngrok.sh` -- the ngrok-specific implementation of all 4 required tunnel functions plus the watchdog and internal helpers.

**Header** (same pattern as other lib/*.sh):
```
#!/usr/bin/env bash
# ============================================
# lib/tunnel/ngrok.sh -- ngrok tunnel provider
#
# Implements: tunnel_start, tunnel_stop,
#   tunnel_status, tunnel_url
#
# Assumes lib/common.sh and lib/config.sh have
# already been sourced by the caller.
# ============================================
```

**Constants** (at top of file, after header):
```bash
TUNNEL_ENV="$CC_TMUX_DIR/tunnel.env"
TUNNEL_LOG="$CC_TMUX_DIR/tunnel.log"
WATCHDOG_PID_FILE="$CC_TMUX_DIR/tunnel-watchdog.pid"
NGROK_API="http://127.0.0.1:4040/api"
```

**Internal helper functions** (prefix with underscore):

1. `_ngrok_is_running()` -- `pgrep -f "ngrok tcp" &>/dev/null`

2. `_ngrok_api_responds()` -- `curl -sf "$NGROK_API/tunnels" &>/dev/null`

3. `_ngrok_persist_address()`:
   - Try `/api/tunnels` first: `curl -sf "$NGROK_API/tunnels"` piped to `jq -r '.tunnels[0].public_url // empty'`
   - If empty, fallback to `/api/endpoints`: `curl -sf "$NGROK_API/endpoints"` piped to `jq -r '.endpoints[0].url // empty'`
   - If still empty, return 1
   - Parse the `tcp://host:port` format using sed and cut (strip `tcp://`, split on `:`)
   - Write to `$TUNNEL_ENV`:
     ```
     # Generated by cc-tmux tunnel -- do not edit manually
     # Last updated: $(date "+%Y-%m-%d %H:%M:%S")
     TUNNEL_HOST="$host"
     TUNNEL_PORT="$port"
     TUNNEL_URL="$public_url"
     TUNNEL_PROVIDER="ngrok"
     ```
   - `chmod 600 "$TUNNEL_ENV"`

4. `_watchdog_is_running()`:
   - Check PID file exists, read PID, validate with `kill -0 "$pid" 2>/dev/null`
   - Also verify process is actually a tunnel watchdog: check `/proc/$pid/cmdline` contains "tunnel" OR just rely on kill -0 (simpler, WSL2 proc can be unreliable)
   - Return 0 if running, 1 if not (clean up stale PID file if process doesn't exist)

5. `_ngrok_stop_watchdog()`:
   - If PID file exists and process is running: `kill "$pid" 2>/dev/null`, `wait "$pid" 2>/dev/null`
   - Remove PID file: `rm -f "$WATCHDOG_PID_FILE"`

6. `_ngrok_start_watchdog()`:
   - Call `_ngrok_stop_watchdog` first (cleanup any existing)
   - Truncate log to last 500 lines before starting: `tail -n 500 "$TUNNEL_LOG" > "${TUNNEL_LOG}.tmp" && mv "${TUNNEL_LOG}.tmp" "$TUNNEL_LOG"` (only if log exists)
   - Start background loop with `nohup bash -c '...' >> "$TUNNEL_LOG" 2>&1 &`
   - Save `$!` to `$WATCHDOG_PID_FILE`
   - The watchdog loop:
     - `backoff=30`, `max_backoff=300`
     - `while true; do sleep 30` (check interval)
     - Health check: if NOT (`pgrep -f "ngrok tcp"` AND `curl -sf "$NGROK_API/tunnels"`):
       - Log: `[timestamp] Tunnel down, restarting (backoff: ${backoff}s)...`
       - Kill existing: `pkill -f "ngrok tcp" 2>/dev/null`, `sleep 2`
       - Start new: `nohup ngrok tcp 22 --log=stdout >> "$TUNNEL_LOG" 2>&1 &`
       - Wait for API: poll with 15 attempts, 1s each
       - If API responds: persist address (inline the curl+jq+write sequence since we can't call bash functions from the nohup subshell), log success, reset backoff to 30
       - If API fails: increase backoff (`backoff=$((backoff * 2))`, cap at max_backoff), log failure with next attempt time
     - IMPORTANT: The watchdog runs in a `nohup bash -c '...'` subshell, so it cannot call the parent shell's functions. The address persistence logic must be inlined (duplicate the curl+jq+write sequence). This is the correct approach -- do not try to export or source functions inside the nohup subshell.

**Public functions** (the 4 required by the provider interface):

7. `tunnel_start()`:
   - Guard: if `_ngrok_is_running` and `_ngrok_api_responds`, log "Tunnel already running", call `tunnel_url`, return 0
   - Check ngrok auth: `if ! ngrok config check &>/dev/null`, log error with hint to run `ngrok config add-authtoken YOUR_TOKEN`, return 1
   - Kill any leftover ngrok: `pkill -f "ngrok tcp" 2>/dev/null`, `sleep 1`
   - Start ngrok: `nohup ngrok tcp 22 --log=stdout > "$TUNNEL_LOG" 2>&1 &`
   - Wait for API: poll `curl -sf "$NGROK_API/tunnels"` up to 15 attempts with 1s sleep each
   - If timeout: `log_error "ngrok failed to start (API not responding after 15s)"`, check if the log contains payment-related errors (grep for "payment\|billing\|upgrade\|paid" in tunnel.log), if so add hint: "TCP tunnels require a payment method on file at https://dashboard.ngrok.com/settings#id-verification (you won't be charged)", kill the process, return 1
   - Call `_ngrok_persist_address` -- if fails, log warning "Could not determine tunnel address"
   - Call `_ngrok_start_watchdog`
   - `log_ok "Tunnel started"`
   - Call `tunnel_url`

8. `tunnel_stop()`:
   - Call `_ngrok_stop_watchdog`
   - `pkill -f "ngrok tcp" 2>/dev/null`
   - `log_ok "Tunnel stopped"`

9. `tunnel_status()`:
   - Initialize: `provider="ngrok"`, `status="down"`, `host=""`, `port=""`, `watchdog_status="stopped"`
   - If `_ngrok_is_running` AND `_ngrok_api_responds`: set `status="connected"`, call `_ngrok_persist_address` (refresh)
   - If `_ngrok_is_running` but NOT `_ngrok_api_responds`: set `status="reconnecting"`
   - Read persisted address from `$TUNNEL_ENV` if file exists (source it, read TUNNEL_HOST, TUNNEL_PORT)
   - Check watchdog: if `_watchdog_is_running`, set `watchdog_status="running"`
   - If first arg is `--json`: output JSON object with all fields using printf
   - Else: human-readable table format:
     ```
     Tunnel Status
     -------------
     Provider:  ngrok
     Status:    connected
     Host:      X.tcp.ngrok.io
     Port:      12345
     Watchdog:  running

     Connect:   ssh -p 12345 user@X.tcp.ngrok.io
     ```
   - Show the Connect line only when status is "connected" and host is non-empty

10. `tunnel_url()`:
    - If `$TUNNEL_ENV` exists, source it
    - If TUNNEL_HOST is non-empty, echo `"${TUNNEL_HOST}:${TUNNEL_PORT}"`, return 0
    - Else return 1

The file should be approximately 200-250 lines. Follow the coding patterns from the existing codebase: 2-space indentation in heredocs, descriptive comments above functions, consistent use of log_* functions.
