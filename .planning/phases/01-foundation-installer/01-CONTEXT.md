# Phase 1: Foundation & Installer - Context

**Gathered:** 2026-03-20
**Status:** Ready for planning

<domain>
## Phase Boundary

One-command installer that handles all dependencies (tmux, openssh-server, ngrok, jq, fail2ban), auto-detects the Windows username, interactively gathers project folders and ngrok auth token, and writes a config system that subsequent phases build on. Must be idempotent and enforce LF line endings via .gitattributes.

Requirements: INST-01, INST-02, INST-03, INST-04, INST-05, INST-06, ROB-04

</domain>

<decisions>
## Implementation Decisions

### Installer UX Flow
- Step-by-step wizard with numbered progress indicators (e.g., `[1/6] Installing dependencies...`)
- Color-coded output: green for success, red for errors, yellow for warnings, blue for prompts
- Each error shows inline with red highlight and a suggested fix action
- Dependency status shown as check/cross per item with automatic install-if-missing
- Support `--yes` flag for non-interactive mode (accept all defaults, skip prompts)
- Installer is a single entry point (`bash install.sh`) that sources lib/ modules

### Project Config Format
- User configuration lives in `~/.cc-tmux/` directory (not in the repo clone)
- Main config: `~/.cc-tmux/config.env` — sourceable KEY=value format, zero dependencies
- Project list: `~/.cc-tmux/projects.conf` — one `name|/path/to/folder` per line (same format as V1 but in a dedicated file)
- Both files are human-readable and hand-editable
- Config generated by installer, consumed by all other scripts

### Installation Directory Structure
- Repository is cloned anywhere (e.g., `~/cc-tmux` or `/mnt/c/Users/.../cc-tmux`)
- Runtime files installed to `~/.cc-tmux/` (lib/, bin/, templates/)
- CLI entry point `cc-tmux` symlinked to `/usr/local/bin/cc-tmux` or PATH exported in .bashrc
- Repo and install dir are separate — repo is source of truth, `~/.cc-tmux/` is runtime
- `cc-tmux update` (Phase 5) will copy updated files from repo to `~/.cc-tmux/`

### ngrok Installation Method
- Install via apt repository (NOT snap — snap fails on WSL2 without systemd as PID 1)
- Add ngrok's official apt signing key and repository
- Prompt user for auth token during install with clear instructions and dashboard URL
- Token stored via `ngrok config add-authtoken` (ngrok's native config)
- If ngrok already installed, skip install and verify config exists (idempotent)
- After setup, run a quick start/stop test tunnel to verify ngrok works

### Line Endings
- `.gitattributes` in repo root forces `* text=auto eol=lf` for all text files
- `*.sh` files explicitly marked as `text eol=lf`
- This prevents Windows Git from converting to CRLF which breaks all bash scripts

### Claude's Discretion
- Exact color codes and formatting details
- Whether to use `tput` or ANSI escape codes for colors
- Error message wording (as long as it's clear and actionable)
- Whether to create `~/.cc-tmux/bin/` or use a flat structure
- How to handle edge case: user has no Documents folder

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `V1/install.sh` — 6-step installer structure can be adapted; username detection via `cmd.exe /C "echo %USERNAME%"` works but needs timeout
- `V1/workspace-init.sh` — Project loop pattern (name|path parsing) can be reused for projects.conf
- `V1/setup-shortcut.ps1` — PowerShell shortcut creation (Phase 6, not this phase)

### Established Patterns
- V1 uses `set -e` for error handling — V2 should add `set -u -o pipefail` for robustness
- V1 detects Windows username via `cmd.exe /C` — V2 needs timeout wrapper (can hang 10-40s per research)
- V1 uses `cat > file << 'EOF'` for file creation — V2 should use template files instead

### Integration Points
- `~/.bashrc` — Auto-attach hook for SSH login (same as V1 but sourcing from ~/.cc-tmux/)
- `/etc/ssh/sshd_config.d/` — SSH config drop-in (Phase 2 builds on this)
- `/etc/sudoers.d/` — Passwordless SSH service management

</code_context>
