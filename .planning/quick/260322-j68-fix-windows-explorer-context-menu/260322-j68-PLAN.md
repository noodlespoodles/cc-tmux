**Create `templates/open-in-cctmux.sh`** -- a new bash script that receives the Windows folder path as $1 and does all the tmux work:

```bash
#!/usr/bin/env bash
# WSL-side helper for Windows Explorer context menu.
# Called by open-in-cctmux.ps1 via wsl.exe.
# Receives Windows folder path as $1.

set -euo pipefail

win_path="${1:-}"
if [[ -z "$win_path" ]]; then
    echo "Usage: open-in-cctmux.sh <windows-folder-path>"
    exit 1
fi

# Convert Windows path to WSL path
wsl_path=$(wslpath "$win_path") || {
    echo "Failed to convert path: $win_path"
    exit 1
}

# Extract folder name for the tmux window name
folder_name=$(basename "$wsl_path")

# Read session name from config (default: work)
CC_TMUX_DIR="$HOME/.cc-tmux"
session_name="work"
if [[ -f "$CC_TMUX_DIR/config.env" ]]; then
    # Source in subshell to get SESSION_NAME
    session_name=$(source "$CC_TMUX_DIR/config.env" 2>/dev/null && echo "${SESSION_NAME:-work}") || session_name="work"
fi

# Check if workspace session exists
if ! tmux has-session -t "$session_name" 2>/dev/null; then
    echo "Workspace not running. Run: cc-tmux start"
    exit 1
fi

# Create new tmux window in the session
tmux new-window -t "$session_name" -n "$folder_name" -c "$wsl_path"
tmux send-keys -t "$session_name:$folder_name" 'powershell.exe' Enter
```

**Rewrite `templates/open-in-cctmux.ps1`** -- thin launcher that passes the folder path as an argument to `wsl.exe`, never using `bash -c`:

```powershell
param([string]$FolderPath)

if (-not $FolderPath) { exit 1 }

# Call the WSL-side helper script with the folder path as an argument.
# By passing arguments individually (not via bash -c), we avoid CRLF
# contamination of the bash command string.
wsl.exe -- bash "$env:HOME/.cc-tmux/templates/open-in-cctmux.sh" $FolderPath
```

IMPORTANT: The PS1 cannot use `$env:HOME` because that's the Windows home, not WSL home. Inside WSL, `~` expands to the WSL user's home. However, `wsl.exe --` runs as the default WSL user, so we need a path that works. The WSL user's home is typically `/home/<username>`. The simplest approach: use `~/.cc-tmux/templates/open-in-cctmux.sh` -- but tilde expansion does not happen when passing args to `wsl.exe -- bash`. Instead, use the `-l` flag on bash so it reads the profile and expands `$HOME`:

Final PS1 content:
```powershell
param([string]$FolderPath)

if (-not $FolderPath) { exit 1 }

# Call WSL-side helper. Using bash -l ensures $HOME is set.
# The helper script path uses $HOME which bash -l resolves.
# We pass the Windows path as a separate argument, avoiding
# any CRLF contamination of bash command strings.
wsl.exe -- bash -l -c "`"$HOME/.cc-tmux/templates/open-in-cctmux.sh`" `"$FolderPath`""
```

Wait -- this still uses `bash -c` which could get CRLF tainted. But the key difference is that the PS1 file on disk has CRLF, but the `-c` argument is constructed at runtime by PowerShell from the param values -- it is NOT read from the file line. PowerShell constructs the command line string in memory. The CRLF issue was that the old PS1 had the entire bash command embedded as a literal string on a single line, and that line ended with `\r\n` from the file encoding, which bash saw as part of the `-c` argument.

Actually, the cleaner approach: do NOT use `bash -c` at all. Instead, use `wsl.exe -- <path-to-script> <arg>`. The trick is getting the script path. Since wsl.exe runs as the default user, we can use:

```powershell
wsl.exe -- ~/.cc-tmux/templates/open-in-cctmux.sh "$FolderPath"
```

But tilde does not expand when passed as an argument to wsl.exe. However, wsl.exe DOES support `~` in its path resolution when using `--`. Let me reconsider.

FINAL APPROACH -- simplest and most robust:

The PS1 will call `wsl.exe` passing the script as a direct executable. Since `wsl.exe -- <command>` runs through the default shell, tilde WILL be expanded by bash:

```powershell
param([string]$FolderPath)
if (-not $FolderPath) { exit 1 }
wsl.exe -- bash -l "$HOME/.cc-tmux/templates/open-in-cctmux.sh" "$FolderPath"
```

No -- `$HOME` in a .ps1 file is PowerShell syntax, not bash. The variable would be empty/wrong.

CORRECT FINAL PS1:
```powershell
param([string]$FolderPath)
if (-not $FolderPath) { exit 1 }
# wsl.exe passes remaining args to the default shell.
# Using single quotes prevents PowerShell from interpolating $HOME.
# bash -l sets up $HOME, then sources the script with the folder as $1.
wsl.exe -- bash -l -c '$HOME/.cc-tmux/templates/open-in-cctmux.sh "$@"' _ $FolderPath
```

This works because:
- PowerShell sees `'$HOME/.cc-tmux/templates/open-in-cctmux.sh "$@"'` as a literal string (single quotes)
- `bash -l -c '...' _ $FolderPath` is the standard pattern for passing args to bash -c
- The `-c` string is built by PowerShell at runtime from string literals, NOT read from a file line with CRLF
- `$HOME` is expanded by bash, not PowerShell
- `$FolderPath` (the Windows path argument) is passed as `$1` to the bash script via `"$@"`

The key insight: CRLF only breaks when the bash command IS the line content from the CRLF file. Here, the `-c` argument is a PowerShell string literal that gets properly terminated. The actual bash logic lives in the .sh file which is deployed with LF endings on the WSL filesystem.

Even simpler alternative that avoids bash -c entirely:

Have `install_context_menu()` write the WSL home path into the PS1 at deploy time (sed substitution), so the PS1 knows the absolute path:

```powershell
param([string]$FolderPath)
if (-not $FolderPath) { exit 1 }
wsl.exe -- /home/username/.cc-tmux/templates/open-in-cctmux.sh $FolderPath
```

**USE THIS APPROACH.** It is the simplest and completely avoids `bash -c`:

1. In `templates/open-in-cctmux.ps1`, use a placeholder `__WSL_HOME__`:
```powershell
param([string]$FolderPath)
if (-not $FolderPath) { exit 1 }
wsl.exe -- __WSL_HOME__/.cc-tmux/templates/open-in-cctmux.sh $FolderPath
```

2. In `install_context_menu()`, after copying the PS1 to `$win_deploy_dir`, run `sed -i "s|__WSL_HOME__|$HOME|g"` on the deployed copy. This bakes in the absolute WSL home path (e.g., `/home/ben`), so PowerShell just calls `wsl.exe -- /home/ben/.cc-tmux/templates/open-in-cctmux.sh C:\Users\Ben\Projects`.

3. `wsl.exe -- /absolute/path/to/script arg` executes the script directly. No `bash -c`, no tilde expansion needed, no CRLF issue. The script file on the WSL side has LF endings.

4. Make `open-in-cctmux.sh` executable (chmod 755) in `step_deploy()` and `install_context_menu()`.
